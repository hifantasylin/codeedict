# 产品需求文档（PRD）

## AI码律 (Codeedict) - 团队协作版

**版本**：v2.1 (含倒计时自动流转机制)
**状态**：草稿 / 提案
**文档日期**：2026-06-29
**目标用户**：采用AI辅助开发的成熟软件研发团队（10-100人规模）
**核心价值定位**：**随叫随到的、高效的AI虚拟研发团队**，将AI编程从“个人副驾驶”升级为“团队协作仪表盘”。

## 1. 产品背景与目标

### 1.1 项目起源
本项目源于对开源项目 **AI码律 (Codeedict) v1.0** 的深度分析与演进设想。v1.0 定位为“面向超级个体的标准化开发工作流”，通过状态机约束、双审查官、历史索引等机制，解决了个人使用AI编码时的“无状态”和“质量不可控”问题。

### 1.2 面临的新挑战
在将其设想应用于**成熟研发团队**时，v1.0的设计暴露出以下关键缺口：
1.  **角色粒度过粗**：将产品、架构、开发、测试等多角色职责合并，无法适应团队专业化分工。
2.  **决策权错位**：赋予AI过多裁决权（如“方向性变更”），缺乏人类（如架构师、Tech Lead）的战略把关。
3.  **协作效率风险**：多角色线性串行工作，若不加优化，将导致流程冗长、重复分析，抵消AI带来的效率。
4.  **知识利用被动**：知识仅为归档，未被主动提炼、复用和植入到日常流程中。
5.  **定制化能力缺失**：审查维度（如七角度）固定，无法承载团队特定的非功能性需求（如安全合规、性能基准）。
6.  **人工依赖瓶颈**：关键决策点仍需人工介入，在人工响应延迟或缺失时，整个流程可能被阻塞。

### 1.3 产品目标
将 **AI码律** 从一个“个人仪表盘”升级为 **团队AI协作操作系统**。核心目标是：
*   **专业化**：引入多角色AI代理，模拟并增强真实研发团队的分工。
*   **高效化**：通过流程裁剪、并行处理、知识复用等机制，确保多角色协作的效率不低于甚至优于单Agent模式。
*   **可控化**：将关键决策点（方案选型、方向变更、复杂逻辑确认）交还给人，AI负责执行、建议和初筛。
*   **可进化**：建立知识反馈回路，让团队的经验和模式成为AI的“肌肉记忆”，持续提升自动化程度。
*   **自动化**：在确保安全的前提下，通过“倒计时自动流转”机制，进一步减少流程对人工即时响应的依赖，实现从“半自动”到“高度自动”的跨越。

## 3. 核心功能需求

### 3.1 多角色协作体系
在v1.0的5个角色基础上，按团队版进行重组和补充，形成以下角色矩阵。该设计借鉴了“三省六部”的制衡思想，实现决策、审核、执行三权分立。

| 角色类别 | 角色名称 | 核心职责 | 关键产出 | 工作模式 |
| :--- | :--- | :--- | :--- | :--- |
| **调度层** | **任务指挥官** | 解析意图、拆分任务、调度角色、管理并行、仲裁资源冲突。 | 任务看板、任务数据包（`task-context.json`） | 自动/规则驱动 |
| **分析层** | **需求分析师** | 澄清模糊需求，拆解为可执行用户故事。 | 结构化需求文档（含验收标准） | 交互式（与PM） |
| | **系统架构师** | 产出技术方案，进行影响分析和备选方案对比。 | 架构方案文档（含决策记录ADR） | 半自动（需架构师确认） |
| **质量层** | **方案评审委员会** | 从多维度（团队可配置）对立审查方案。 | 带风险等级的评审报告 | 自动+异步人工抽检 |
| | **质检员 (QA)** | 运行自动化测试、静态扫描、检查覆盖率。 | 质量门禁通过/失败报告 | 全自动（CI集成） |
| | **代码评审员** | 审查代码与方案一致性、规范、可读性。 | 带建议的差异审查报告 | 自动+Tech Lead确认 |
| **知识层** | **知识管理员** | 提炼归档方案，建立模式库，主动推送历史经验。 | 团队模式库、历史案例索引 | 自动/定期更新 |
| **执行层** | **开发程序员** | 按拆解任务和方案进行编码实现和自测。 | 代码变更集（Commit） | 半自动（开发主导） |

### 3.2 效率保障机制
为解决多角色带来的效率问题，内置以下四大机制：

1.  **上下文传递与增量分析**
    *   **功能**：任务指挥官在委派时，传递一个包含所有前序产出摘要的**任务数据包**，禁止角色从零分析。
    *   **需求**：角色启动时，必须首先读取并理解数据包，其产出必须基于数据包进行**增量式补充**。

2.  **流水线并行与智能闸口**
    *   **功能**：定义三种闸口类型，由任务指挥官根据风险等级自动选择：
        *   **自动闸口**：低风险变更，通过AI审查和自动化测试后，**自动放行**。
        *   **异步闸口**：中等风险变更，通过IM向Tech Lead发送摘要和审批链接，设定**2小时**响应时限，不阻塞CI。
        *   **同步闸口**：高风险变更（架构调整），触发**正式评审会议**，需要架构师或委员会确认。
    *   **需求**：支持团队自定义闸口规则和风险等级。

3.  **智能缓存与模式复用**
    *   **功能**：知识管理员建立的模式库（如“登录安全审查清单”），可被评审委员会和代码评审员**直接引用**，进行快速核对，而非每次重新推理。
    *   **需求**：审查报告需明确指出**新发现的问题**和**已由模式库覆盖的问题**。

4.  **决策记录与可追溯性**
    *   **功能**：所有AI的裁决、建议和用户的审批操作，均以结构化的**决策记录（ADR）** 形式归档到团队知识库。
    *   **需求**：支持按需求ID、时间、角色、决策类型进行检索和审计。

### 3.3 任务复杂度分级与流程动态裁剪
系统根据任务复杂度，自动选择处理路径，避免“一刀切”：

| 任务类型 | 典型示例 | 核心特征 | 自动裁剪后的轻量化流程 |
| :--- | :--- | :--- | :--- |
| **小Bug / 热修复** | “登录页按钮文案错位” | 影响范围小、根因明确、修复方案单一。 | 需求分析 → 程序员 → 质检员 → 代码评审员 → 归档（跳过架构师、方案评审） |
| **大需求 / 新功能** | “增加用户积分系统” | 涉及多模块、需设计新方案、有明确验收标准。 | 需求分析师 → 系统架构师 → 方案评审委员会 → 程序员 → 质检员 → 代码评审员 → 归档（触发异步闸口） |
| **复杂问题 / 架构级变更** | “重构用户认证模块” | 影响范围广、跨多服务、有重大技术或业务风险。 | **完整流程**，并启用：多备选方案生成、渐进式披露决策、分阶段实施计划、多同步闸口。 |

### 3.4 人工决策点与倒计时自动流转机制（核心增强）

#### 3.4.1 设计目标
在必须保留的人工决策环节，引入**可配置的倒计时自动流转机制**，将人工参与从“必需”转变为“可选干预”，在保证安全的前提下，极大提升流程的自动化程度和鲁棒性。

#### 3.4.2 核心机制
*   **触发**：当流程进入特定“待人工决策”状态时，系统自动启动一个可视化的倒计时器。
*   **干预**：在倒计时结束前，相关责任人可通过看板或IM通知进行人工决策（如“准奏”、“驳回”）。
*   **超时**：若倒计时结束且无人干预，系统将根据预设的“默认自动策略”安全地推进流程。
*   **记录**：所有自动决策均被显著标记并记录在审计日志中。

#### 3.4.3 适用场景、默认策略与风险控制

| 流程环节 | 人工决策点 | 倒计时（建议） | 超时后的默认自动策略 | 风险控制与备注 |
| :--- | :--- | :--- | :--- | :--- |
| **方案审核** | 门下省对高风险或“封驳”方案的“御批”确认。 | 3秒 | **自动“准奏”**（放行），标记为“自动通过”。 | **仅适用于**AI评估为**低风险**或**已多次审核通过**的方案。高风险方案强制人工，不启用倒计时。 |
| **任务派发** | 对尚书省推荐的任务分配方案做最终确认。 | 3秒 | **自动采纳系统推荐的分配方案**（基于负载和绩效）。 | 需确保推荐算法足够智能，可设置资源水位线，超过则升级人工。 |
| **代码审查** | Tech Lead对AI报告中的“方向性变更”做最终裁决。 | 5秒 | **自动采纳AI评审员的建议**（“驳回”或“需修改”），并归档依据。 | 仅对变更影响范围明确的PR启用。 |
| **跨模块影响审批** | 受影响模块的Code Owner审批。 | 10秒 | **暂停流程**，自动向备份负责人或团队主管发送**升级通知**。 | **高风险环节，不放行**。采用“升级通知”策略，确保有人工介入。 |

#### 3.4.4 用户交互与体验
*   **看板可视化**：在任务详情页清晰显示倒计时（如“等待御批：3...2...1...”）。
*   **IM即时通知**：倒计时开始时，通过飞书、钉钉等向责任人发送带“快速审批”按钮的预警通知，提供最后的介入窗口。
*   **事后一键覆盖**：在自动决策后的短暂窗口期（如5分钟内），提供“撤销并介入”按钮，允许错过时机的决策者拉回控制权。

#### 3.4.5 配置与管理
*   **可配置性**：所有倒计时时长和超时默认策略均为**可配置项**（如在`openclaw.json`中定义）。
*   **风险分级**：支持按**任务类型**（Bug/新功能/架构变更）、**影响范围**（模块/核心服务）设置不同的倒计时规则。
*   **审计与报告**：自动决策在“奏折”和审计日志中用“⚠️ 自动御批”等醒目标记，确保全程可追溯。

## 5. 用户流程示例：新增“用户积分”功能（含自动流转）

1.  **PM**：在IDE或Web端提出需求 → 任务指挥官创建任务数据包，委派给需求分析师。
2.  **需求分析师 (AI)**：与PM交互澄清规则，产出结构化需求文档。
3.  **系统架构师 (AI)**：产出技术方案，提交给**架构师**异步审批（闸口）。
4.  **方案评审委员会 (AI)**：多角度审查方案，产出评审报告。
5.  **自动流转触发**：此方案经AI评估为“中等风险”，进入“待架构师确认”状态。系统启动**3秒倒计时**，并通过IM通知架构师。架构师若在3秒内未响应，系统**默认采纳AI方案**，流程自动继续。
6.  **任务指挥官 (AI)**：将批准后的方案拆解为子任务，派发给开发程序员。
7.  **开发程序员 (AI + 人)**：AI辅助生成代码，提交PR。
8.  **质检员 (AI)** & **代码评审员 (AI)**：自动运行测试和生成审查报告。
9.  **二次自动流转**：PR审查进入“待Tech Lead确认”状态，启动**5秒倒计时**。Tech Lead若超时未处理，系统**自动采纳AI评审员的建议**（如通过或请求修改）。
10. **知识管理员 (AI)**：任务归档后，提炼关键决策，更新团队模式库。所有“自动通过”的决策在奏折中被显著标记。

## 6. 部署与集成场景

### 6.1 核心运行模式：事件驱动
系统并非24小时持续运行的守护进程，而是**完全事件驱动**的：
*   **触发事件**：开发人员发起新任务、PR被创建或更新、人工决策超时等。
*   **运行方式**：以“任务”为单位启动一个独立的处理实例，处理完成后终止。
*   **资源消耗**：仅在任务处理期间消耗计算资源。等待人工决策时处于休眠状态。

### 6.2 首选集成场景：CodeBuddy 自定义Agent
*   **触发入口**：在CodeBuddy聊天框中通过斜杠命令（如 `/codeedict 修复登录页按钮错位`）唤醒。
*   **上下文获取**：CodeBuddy自动提供当前项目的文件结构、打开的文件内容、错误日志等。
*   **结果呈现**：处理结果直接流式返回到聊天界面。
*   **实现方式**：将整个系统打包为**一组CodeBuddy子代理（Sub-agents）**，通过项目级或用户级配置文件定义，并可打包成**插件（Plugin）** 在团队内部分发。

### 6.3 其他集成可能
*   **VS Code (GitHub Copilot)**：通过Custom Instructions或Agent扩展点集成。
*   **JetBrains IDE**：通过**Agent Client Protocol (ACP)** 协议集成。
*   **独立Web服务**：作为独立的团队协作平台。

## 7. 约束、风险与假设

### 7.1 约束
*   目前主流大模型（GPT-4、Claude 3.5等）的上下文窗口和推理能力需足以处理复杂的多角色调度和长文档分析。
*   团队需具备一定的DevOps和基础设施（如Git、CI）基础。

### 7.2 关键风险与缓解
*   **风险：AI审查的“幻觉放大”** → **缓解**：引入人工抽检机制；建立模式库作为“事实锚点”；关键决策强制人工。
*   **风险：流程过度设计导致采用成本高** → **缓解**：从轻量级场景（如CodeBuddy Agent）切入，支持渐进式采用。
*   **风险：自动流转导致错误决策放大** → **缓解**：实施**严格的风险分级**（高风险不放行）、**全流程审计**、提供**事后人工覆盖**按钮。
*   **风险：数据安全与隐私** → **缓解**：支持私有化部署和本地模型；敏感数据脱敏处理。

### 7.3 假设
*   团队成员对AI辅助编码有基本认知和接受度。
*   团队愿意投入少量配置成本以换取长期效率提升。
*   团队能定义清晰的任务风险等级和对应的自动流转策略。
